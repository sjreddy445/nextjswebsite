    Part 2 | A Glimpse of JavaScript in the backend – nodeJS

    Welcome back Geeks. In part 2 of the tech-blog on nodeJS, let’s see how it works, what are its benefits and its demerits.

    How nodeJS works?

    ■    Client pushes the request (events) to node server.
    ■    All events are pushed to event loop (event loop is maintained by socket handler, tunnels
    and interface cards)
    ■    Events in event loop meet with event executer where they are executed one by one.
    ■    If event is asking for I/O operation they are pushed to I/O queue else response is returned
    to the client.
    ■    I/O handler entertain I/O requests one by one from I/O loop and taks with corresponding
    interface e.g. DMA(direct memory access), Database(like ODBC), file system.
    ■    After completion of I/O request response event is pushed back to event loop and event
    handler sends response to client.

    Pros and Cons of nodeJS

    Pros

    		 Cons
    1. Asynchronous event driven IO helps concurrent request handling. 1. nodeJS doesn’t provide scalability. One CPU is not going to be enough; the platform provides no ability to scale out to take advantage of the multiple cores commonly present in today’s server-class hardware.
    2. Uses JavaScript, which is easy to learn. 2. Dealing with relational database is a pain if you are using nodeJS.
    3. Share the same piece of code with both server and client side. 3. Every time using a callback end up with tons of nested callbacks.
    4. NPM, the Node Packaged Modules has already become huge, and still growing. 4. Without diving in depth of JavaScript if someone starts nodeJS, he may face conceptual problem.
    5. Active and vibrant community, with lots of code shared via github, etc. 5. nodeJS is not suited for CPU-intensive tasks. It is suited for I/O stuff only (like web servers).

    6. You can stream big files.
